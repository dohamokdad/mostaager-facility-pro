# Mostaager Facility Pro - Analytics Hub Technical Specification

## 1. الهدف
إنشاء لوحة تحليلات داخلية في إضافة `Mostaager Facility PRO` لعرض مؤشرات الأداء التشغيلية للملاك والمدراء مع بيانات فورية عن:
- نسبة الإشغال
- الفواتير المستحقة والمدفوعة
- طلبات الصيانة المفتوحة
- أداء المحفظة المالية لكل مبنى

## 2. نطاق العمل
### 2.1 ضمن النطاق
- إضافة صفحة لوحة إدارة Dashboard في الإضافة
- إنشاء REST API خاصة بتحليلات Mostaager
- عرض مخططات ورسوم بيانية داخلية
- دعم فلاتر زمنية وفلترة حسب المبنى
- دعم صلاحيات عرض خاصة بالأدوار المناسبة

### 2.2 خارج النطاق
- تطبيق موبايل مستقل
- تكامل CRM خارجي كامل
- إنتاج تقارير PDF متقدمة

## 3. المتطلبات الوظيفية
### 3.1 اللوحة الرئيسية
عرض مؤشرات سريعة (KPI cards):
- عدد المباني
- نسبة الإشغال الإجمالية
- إجمالي المبالغ المستحقة
- إجمالي المبالغ المدفوعة
- عدد طلبات الصيانة المفتوحة
- إجمالي رصيد المحفظة المالية

### 3.2 بيانات المباني
عرض جدول ملخص لكل مبنى يحتوي على:
- اسم المبنى أو العنوان
- عدد الوحدات
- نسبة الإشغال
- رصيد المحفظة
- الفواتير المتأخرة
- طلبات الصيانة المفتوحة

### 3.3 فلاتر زمنية
- اليوم
- الأسبوع
- الشهر
- الربع
- السنة

### 3.4 تنبيهات حرجة
- فواتير متأخرة
- صيانة مفتوحة لأكثر من X يوم
- انخفاض رصيد المحفظة دون الهدف

## 4. بنية البيانات
### 4.1 مصادر البيانات الأساسية
- `ms_buildings`
- `ms_invoices`
- `ms_maintenance_requests`
- `ms_maintenance_comments`
- `ms_building_wallet`

### 4.2 دوال مقترحة في `Mostaager_DB`
- `get_buildings_analytics(): array`
- `get_invoice_summary(array $args): array`
- `get_maintenance_summary(array $args): array`
- `get_building_occupancy(int $building_id): array`

## 5. واجهة برمجة التطبيقات (API)
### 5.1 نقاط النهاية
- `GET /wp-json/ms/v1/analytics/summary`
  - بيانات المؤشرات الرئيسية
- `GET /wp-json/ms/v1/analytics/buildings`
  - بيانات الملخص لكل مبنى
- `GET /wp-json/ms/v1/analytics/invoices`
  - بيانات الفواتير حسب الحالة والمبنى
- `GET /wp-json/ms/v1/analytics/maintenance`
  - بيانات طلبات الصيانة حسب الحالة والمبنى

### 5.2 صلاحيات
- فقط للمستخدمين الذين يملكون:
  - `manage_options`
  - أو `ms_view_dashboard`

## 6. واجهة المستخدم
### 6.1 مكونات الواجهة
- بطاقات KPI
- مخطط خطي للإيرادات والمدفوعات
- مخطط دائري لنسبة الإشغال
- جدول مباني مع ملخص الحالة
- قائمة تنبيهات حرجة

### 6.2 مبادئ UX
- تصميم RTL
- تجربة Mobile First
- تحميل بيانات ديناميكي عبر AJAX
- طلاء CSS/JS خاص باللوحة فقط

## 7. الأداء والتحسين
### 7.1 الكاش
- استخدام `transients` للنتائج الثقيلة
- تحديث الكاش عند:
  - إنشاء أو تعديل فاتورة
  - تغيير حالة فاتورة
  - إنشاء أو تعديل طلب صيانة
  - تعديل رصيد المحفظة

### 7.2 استعلامات محسنة
- تجنب `SELECT *`
- استخدام `GROUP BY`, `SUM`, `COUNT`
- تجميع البيانات عبر دوال `Mostaager_DB`

## 8. خطة التنفيذ
### المرحلة 1: التحضير
- تحديد البيانات المطلوبة
- تصميم API وواجهات المستخدم
- رسم مخطط Dashboard

### المرحلة 2: التنفيذ الخلفي
- تطوير وظائف تحليلية في `includes/class-mostager-db.php`
- إنشاء REST API endpoints في `includes/api.php`
- إعداد كاش `transient`

### المرحلة 3: تنفيذ الواجهة
- بناء صفحة لوحة تحكم داخل admin
- استخدام Chart.js أو مكتبة خفيفة للرسوم
- بناء فلاتر الزمن والمباني

### المرحلة 4: الاختبار والتسليم
- اختبار صلاحيات الوصول
- اختبار الأداء والتحميل
- اختبار التوافق مع WPML و Houzez

## 9. النتائج المتوقعة
- لوحة تحكم إدارية قوية لقياس أداء الأداء العقاري
- تحويل الموقع من أداة عرض عقارات إلى أداة إدارة تشغيلية
- دعم قرار العمل والتحويل داخل الموقع نفسه
